### 第1步：检查 Python 和依赖

```bash
# 注意：替换为你的实际路径
cd c:\cursor_Xiangmu\taidi_cup_B\financial-assistant\chatbot_financial_statement

# 1. 检查 Python 版本
python --version
# 需要 3.10+，如果不是，请下载安装：https://www.python.org/downloads/

# 2. 检查 streamlit 是否安装
pip show streamlit

# 3. 如果未安装或版本不对，安装依赖
pip install -r requirements.txt

# 💡 提示：如果下载慢，使用国内镜像
pip install -r requirements.txt -i https://pypi.tuna.tsinghua.edu.cn/simple
```

**如果遇到问题**：
- ❌ `'python' 不是内部或外部命令` → Python 未安装或未添加到 PATH
  - **解决**：重新安装 Python，勾选 "Add Python to PATH"
- ❌ `pip 不是内部或外部命令` → 虚拟环境未激活
  - **解决**：先运行 `venv\Scripts\activate`（Windows）或 `source venv/bin/activate`（Mac/Linux）
- ❌ 某个包安装失败（如 `psycopg2`）→ 缺少编译工具
  - **解决**：`pip install wheel` 然后重试，或使用预编译版本

## ⚠️ 常见问题（基于实际遇到的坑）

### ❌ 问题1：PostgreSQL 连接报编码错误

**现象**（我遇到的）：
```
❌ PostgreSQL连接失败: 'utf-8' codec can't decode byte 0xd6 in position 61: invalid continuation byte
```

**原因**：
- Docker 创建的 PostgreSQL 默认使用 `UTF8` 编码
- 但项目数据是 **GBK 编码**的中文财报数据
- 导致解码失败

**解决方案 A：重建数据库为 GBK 编码（推荐）**

1. **停止并删除现有容器**：
   ```bash
   docker-compose down
   ```

2. **删除旧数据库卷**：
   ```bash
   # 删除 PostgreSQL 数据卷
   docker volume rm financial-assistant_postgres_data
   ```

3. **修改 docker-compose.yml**（如果需要永久解决）：
   ```yaml
   postgres:
     image: postgres:15-alpine
     container_name: postgres2
     ports:
       - "${DB_PORT}:5432"
     environment:
       POSTGRES_DB: ${DB_NAME}
       POSTGRES_USER: ${DB_USER}
       POSTGRES_PASSWORD: ${DB_PASSWORD}
       POSTGRES_INITDB_ARGS: "--encoding=GBK"
     volumes:
       - postgres_data:/var/lib/postgresql/data
   ```

4. **重新启动**：
   ```bash
   docker-compose up -d
   ```

5. **验证编码**：
   ```bash
   # 进入容器
   docker exec -it postgres2 psql -U postgres -d financial_db -c "\l"
   # 查看编码是否为 GBK 或 UTF8（UTF8 通常也能处理 GBK 数据）
   ```

**解决方案 B：在应用层处理编码（临时方案）**

如果不想改 Docker 配置，可以在 Python 连接时指定编码：

```python
import psycopg2
conn = psycopg2.connect(
    host='localhost',
    port=5432,
    dbname='financial_db',
    user='postgres',
    password='postgres',
    options='-c client_encoding=GBK'
)
```

> **💡 经验总结**：如果数据是 GBK 编码，PostgreSQL 用 UTF8 也可能正常（PostgreSQL 能自动转换）。但如果报错，尝试用 GBK 编码重建数据库。

### ❌ 问题2：MongoDB 认证失败

**现象**（我遇到的）：
```
❌ MongoDB连接失败: Authentication failed., full error: {'ok': 0.0, 'errmsg': 'Authentication failed.', 'code': 18, 'codeName': 'AuthenticationFailed'}
```

**原因**：
- Docker 启动 MongoDB 时，会自动使用环境变量创建 root 用户
- 但如果容器已经存在（之前运行过），环境变量不会重新创建用户
- 导致密码不匹配

**解决方案 A：彻底重建（最可靠）**

1. **停止并删��容器和数据卷**：
   ```bash
   docker-compose down
   docker volume rm financial-assistant_mongo_data
   ```

2. **重新启动**：
   ```bash
   docker-compose up -d
   ```

3. **验证登录**：
   ```bash
   # 进入容器
   docker exec -it mongodb2 bash

   # 连接 MongoDB
   mongosh -u admin -p admin --authenticationDatabase admin

   # 如果成功，说明账号密码正确
   ```

**解决方案 B：手动重置 MongoDB 管理员密码**

如果数据很重要不想删除，可以：

1. **以无认证模式启动 MongoDB**：
   ```bash
   # 停止容器
   docker-compose down

   # 手动启动（无认证）
   docker run -d --name mongodb_temp \
     -p 27017:27017 \
     -v c:\cursor_Xiangmu\taidi_cup_B\financial-assistant\chatbot_financial_statement\mongo_data:/data/db \
     mongo:6.0 --noauth

   # 进入容器创建用户
   docker exec -it mongodb_temp mongosh

   # 在 mongosh 中执行：
   use admin
   db.createUser({
     user: "admin",
     pwd: "admin",
     roles: [{ role: "root", db: "admin" }]
   })
   ```

2. **停止临时容器，恢复原容器**：
   ```bash
   docker stop mongodb_temp
   docker-compose up -d
   ```

**解决方案 C：修改连接配置（不推荐）**

如果只是测试，可以在 `.env` 中禁用认证（仅开发环境）：

```ini
# 不推荐生产使用！
MONGO_DB_USER=
MONGO_DB_PASSWORD=
```

然后在 `docker-compose.yml` 中移除环境变量配置（但这样不安全）。

### ❌ 问题3：Docker 容器启动失败

**现象**：`docker-compose up -d` 后，`docker ps` 看不到容器

**解决步骤**：

1. **检查 Docker Desktop 状态**：
   - Windows：任务栏右键 Docker 图标 → "Show Dashboard"
   - 如果没运行，从开始菜单启动

2. **查看详细日志**：
   ```bash
   docker-compose logs
   # 或查看特定服务
   docker-compose logs postgres
   docker-compose logs mongodb
   ```

3. **常见错误和解决**：

   **错误1**：`port is already allocated`
   ```bash
   # 原因：5432 或 27017 端口被占用
   # 解决：查看占用进程并杀死
   netstat -ano | findstr :5432  # Windows
   lsof -i :5432  # Mac/Linux

   # 杀死占用进程（Windows）
   taskkill /PID <PID> /F
   ```

   **错误2**：`permission denied`（Linux/Mac）
   ```bash
   # 当前用户没有 Docker 权限
   sudo usermod -aG docker $USER
   # 然后退出终端重新登录
   ```

   **错误3**：`pull access denied` 或 `image not found`
   ```bash
   # 原因：Docker 镜像拉取失败（网络问题）
   # 解决：配置 Docker 镜像加速器
   # Windows/Mac：Docker Desktop → Settings → Docker Engine → 添加镜像加速器
   # 示例配置：
   {
     "registry-mirrors": [
       "https://docker.mirrors.ustc.edu.cn",
       "https://hub-mirror.c.163.com"
     ]
   }
   ```


**解决**：

```bash
# 1. 查看占用端口的进程

# Mac/Linux：
kill -9 <PID>

# 然后访问 http://localhost:8502
```

### ❌ 问题7：setup.py 运行失败或卡住

**现象**：`python setup.py` 报错或无响应

**可能原因和解决**：

1. **API 密钥无效**：
   ```bash
   # 检查 .env 中的 OPENAI_API_KEY 是否正确
   # 在浏览器登录 https://platform.deepseek.com/ 查看 API Keys
   ```

2. **网络问题**：
   ```bash
   # 测试 API 连通性
   curl https://api.deepseek.com/v1/chat/completions
   # 或 Windows：
   Invoke-WebRequest https://api.deepseek.com/v1/chat/completions
   ```

3. **内存不足**：
   - setup.py 需要处理 200+ 家公司数据
   - **解决**：使用小数据版本 `--preprocess v3`（100 家公司）

4. **磁盘空间不足**：
   - 需要至少 2GB 可用空间
   - 检查：`dir`（Windows）或 `df -h`（Linux）

5. **超时问题**：
   ```bash
   # 使用 --force True 强制重新处理
   python setup.py --preprocess v3.2 --force True --openai True --vectordb chromadb

   # 或使用更快的配置
   python setup.py --preprocess v3.2 --force False --openai True --vectordb chromadb
   ```
